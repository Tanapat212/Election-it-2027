# เช็คลิสต์ก่อนขึ้น Server จริง

## ✅ เพิ่งแก้ให้แล้วในไฟล์ล่าสุด (ช่องโหว่ความปลอดภัยที่เจอจริง)

1. **จำกัดจำนวนครั้งการ login** — เดิมไม่มีการจำกัดเลย ใครก็สามารถลองสุ่มรหัสผ่านซ้ำๆ ไม่จำกัดครั้งได้ (brute-force) ตอนนี้จำกัดไว้ **ไม่เกิน 8 ครั้ง ต่อ 5 นาที ต่อ 1 IP** สำหรับหน้า login ทั้งแอดมินและผู้มีสิทธิ์
2. **จำกัด request รวมทั้งระบบ** — ไม่เกิน 300 ครั้ง/นาที/IP กัน bot ยิงถล่ม
3. **ซ่อนรายละเอียด error ภายใน** — เดิมถ้าเซิร์ฟเวอร์ error (เช่น ต่อฐานข้อมูลไม่ติด) ข้อความ error ดิบๆ จากฐานข้อมูลจะหลุดไปให้ผู้ใช้เห็นได้ ตอนนี้ตั้งค่า production ไว้ให้โชว์แค่ "เกิดข้อผิดพลาดที่เซิร์ฟเวอร์" แทน
4. **กันรันด้วย JWT_SECRET ค่า default** — ถ้าลืมตั้งรหัสลับใหม่ก่อนขึ้น production เซิร์ฟเวอร์จะ**ปฏิเสธไม่ยอมรัน**เลยทันที (ดีกว่าปล่อยให้รันด้วยความเสี่ยงที่คนนอกปลอมโทเคนได้)
5. **จำกัดชนิดไฟล์ที่อัปโหลดได้** — เดิมไม่เช็คเลยว่าไฟล์ที่อัปโหลด (รูปผู้สมัคร/ไฟล์นโยบาย) เป็นไฟล์อะไรจริงๆ ใครจะอัปโหลดไฟล์อันตรายสวมชื่อเป็นรูปก็ได้ ตอนนี้จำกัดไว้แค่ JPG/PNG/WEBP (และ PDF สำหรับไฟล์นโยบาย) เท่านั้น
6. **เพิ่ม security headers พื้นฐาน** (@fastify/helmet) กัน clickjacking และ header-based attack ทั่วไป

⚠️ **ข้อ 1, 5, 6 ต้องรัน `npm install` ใหม่ในโฟลเดอร์ backend** เพราะเพิ่ม 2 patackage ใหม่ (`@fastify/rate-limit`, `@fastify/helmet`)

## 🔴 ต้องทำเองก่อนขึ้นจริง (ผมแก้ในโค้ดให้ไม่ได้ เพราะเป็นเรื่องบัญชี/โครงสร้างพื้นฐานของคุณ)

1. **เปลี่ยนรหัสผ่าน admin เริ่มต้น** — ตอนนี้ยังเป็น `ChangeMe123!` (ค่า default ใน `.env.example`) **ต้องตั้ง `SEED_ADMIN_PASSWORD` เป็นรหัสใหม่ในไฟล์ `.env` จริงก่อนรัน seed บนเซิร์ฟเวอร์จริง** หรือถ้า seed ไปแล้วด้วยรหัสเดิม ให้เข้าไปเปลี่ยนรหัสที่แท็บ "บัญชีของฉัน" ทันที
2. **ตั้ง `JWT_SECRET` ใหม่เป็นค่าสุ่มยาวๆ** ในไฟล์ `.env` บนเซิร์ฟเวอร์จริง (ห้ามใช้ค่าตัวอย่างจาก `.env.example` เด็ดขาด) — สุ่มได้จากเว็บ เช่น https://generate-secret.vercel.app/32 หรือรันคำสั่ง `openssl rand -hex 32`
3. **ตั้ง `NODE_ENV=production`** ในไฟล์ `.env` ตอน deploy จริง (มีผลกับข้อ 3 ด้านบน ที่ซ่อนรายละเอียด error)
4. **ตั้งรหัสผ่านฐานข้อมูล MySQL ให้มีรหัสผ่านจริง** — ตอนพัฒนาในเครื่องใช้ root ไม่มีรหัสผ่านได้ แต่บนเซิร์ฟเวอร์จริงต้องตั้งรหัสผ่าน MySQL ด้วยเสมอ แล้วอัปเดต `DB_PASSWORD` ใน `.env`
5. **ใช้ HTTPS** — ต้องมี SSL certificate (เช่นผ่าน Let's Encrypt ฟรี หรือที่ host ให้มาเลย) ไม่งั้นรหัสผ่านที่พิมพ์ตอน login จะวิ่งผ่านเน็ตแบบไม่เข้ารหัส ใครดักจับแพ็กเก็ตกลางทางอ่านได้เลย — ข้อนี้สำคัญมากเพราะระบบนี้มีการกรอกรหัสผ่าน
6. **สำรองข้อมูล (backup) ฐานข้อมูลเป็นประจำ** — โดยเฉพาะช่วงวันเลือกตั้งจริง แนะนำตั้ง cron/schedule export ฐานข้อมูลอัตโนมัติ (หรืออย่างน้อย export มือก่อน-หลังวันเลือกตั้ง)

## 🟡 เรื่องที่ควรพิจารณา (ไม่ใช่บั๊ก แต่เป็นข้อจำกัดที่ควรรู้ไว้)

1. **รันด้วย `npm run dev` ไม่เหมาะกับ production** — ตอนขึ้นเซิร์ฟเวอร์จริง frontend ควรรันด้วย `npm run build` แล้วตามด้วย `npm run start` (โหมด production ของ Next.js จะเร็วและเบากว่า `next dev` มาก) ส่วน backend ใช้ `npm run start` แทน `npm run dev` เช่นกัน (ตัว `dev` ใช้ `--watch` ซึ่งไม่จำเป็นตอน production)
2. **ควรมี process manager** เช่น PM2 หรือ systemd คอย restart backend/frontend อัตโนมัติถ้า crash ไม่งั้นเว็บล่มแล้วต้องมานั่ง manual restart เอง
3. **ต้องมี reverse proxy** (เช่น Nginx) หน้าเว็บ เพื่อรวม frontend (port 3000) กับ backend (port 4000) ให้อยู่โดเมนเดียวกัน และจัดการ HTTPS ให้
4. **XAMPP ไม่เหมาะกับ production จริงจัง** — XAMPP ออกแบบมาไว้พัฒนาในเครื่องเป็นหลัก ถ้าจะขึ้นเซิร์ฟเวอร์จริงแนะนำใช้ MySQL/MariaDB แบบติดตั้งปกติ หรือใช้ managed database service แทน
5. **ยังไม่มีระบบสำรอง/กู้คืนอัตโนมัติเผื่อระบบล่มกลางทางโหวต** — ถ้ากังวลเรื่องนี้มากๆ (เช่น ใช้แข่งจริงวันเดียว คนพร้อมกันเยอะ) แนะนำซ้อมระบบเต็มรูปแบบ (dry-run) ก่อนวันจริงอย่างน้อย 1 รอบ

## 🟢 พร้อมแล้ว (ไม่ต้องทำอะไรเพิ่ม)

- โครงสร้างฐานข้อมูลและ business logic (เปิด/ปิดโหวต, นับคะแนน, กันโหวตซ้ำ) ผ่านการรีวิวแล้ว
- รหัสผ่านทุกบัญชีเก็บแบบ hash (bcrypt) ไม่เก็บ plain text
- มี audit log บันทึกการกระทำของแอดมินทุกคน ป้องกันไม่ให้ลบได้
- จำกัดเวลาการ login อัตโนมัติ (แอดมิน 30 นาที / ผู้โหวต 15 นาที)
- ระบบ responsive ใช้งานได้ทั้งจอคอม/มือถือ
